## 1. Цель и постановка задачи

Разработать и сравнить несколько лёгких (nano/small) нейросетевых детекторов объектов для задачи обнаружения дефектов пластиковых бутылок на производственной линии. Итог — обоснованный выбор модели по совокупности метрик точности, скорости и размера, плюс работающий прототип, демонстрирующий детекцию в реальном времени с подсчётом дефектов.

**Задачи:**
1. Подготовить унифицированный датасет дефектов бутылок на основе open-source источников.
2. Обучить 4 модели семейства YOLO (v8n, v8s, v11n, v11s) в одинаковых условиях.
3. Сравнить модели по точности (mAP, Precision, Recall), скорости (FPS) и компактности (params, FLOPs, MB).
4. Оценить устойчивость к смене распределения (OOD-тест на другом датасете).
5. Реализовать real-time демо с камеры/видео с подсчётом дефектов по классам.

## 2. Дефекты, которые детектируем

Целевая унифицированная таксономия классов (финальный список подтверждается после анализа меток датасета):

| Класс | Описание |
|---|---|
| `deformation` | Деформация корпуса бутылки (вмятины, перекос) |
| `label_defect` | Отрыв/смещение/отсутствие этикетки |
| `cap_defect` | Отсутствующая/повреждённая/неправильно закрытая крышка |
| `fill_level` | Неправильный уровень налива (недолив/перелив) |
| `no_defect` (опционально) | Годная бутылка — для подсчёта общего числа |

Фактическое множество классов зависит от выбранного датасета. Маппинг исходных меток в унифицированную таксономию описан в `configs/class_mapping.yaml`.

## 3. Данные

**Основной датасет:** `spark-intelligence/bottle-defect-detection` (Roboflow Universe). Используется для train/val/test.

**OOD-тест:** `thesis-work-l9lqp/bottle-defect-3` (Roboflow Universe). Только для итогового тестирования обобщающей способности лучшей модели.

**Загрузка:** через `roboflow` Python SDK. Требуется бесплатный API-ключ (берётся из env `ROBOFLOW_API_KEY`).

**Препроцессинг:**
- Формат YOLO (txt-аннотации).
- Изображения ресайзятся моделью при обучении (imgsz=640).
- Разбиение train/val/test = 70/20/10 (если исходный датасет не имеет деления — делаем сами с фиксированным seed=42).
- Аугментации — дефолтные Ultralytics (mosaic, mixup, hsv, flip).

**Если в исходном датасете классы не совпадают с целевой таксономией** — скрипт `scripts/remap_classes.py` переименовывает/объединяет классы согласно `class_mapping.yaml`. Если датасет содержит только часть целевых классов — работаем с тем, что есть, это документируется в thesis-отчёте.

## 4. Модели для сравнения

| Модель | Параметры (M) | Ожидаемый FPS (GPU) | Роль |
|---|---|---|---|
| YOLOv8n | ~3.2 | ~300+ | Baseline |
| YOLOv8s | ~11.2 | ~150+ | «Средний» вариант v8 |
| YOLOv11n | ~2.6 | ~350+ | Новая архитектура, nano |
| YOLOv11s | ~9.4 | ~170+ | Новая архитектура, small |

**Почему именно эти:** YOLOv8 — индустриальный стандарт 2023-2024, YOLOv11 — текущее поколение Ultralytics (2024-2025). Обе архитектуры доступны через один и тот же SDK, что гарантирует честность сравнения (один пайплайн, одинаковые аугментации).

**Единые условия обучения для всех моделей:**
- `imgsz=640`, `epochs=100`, `patience=20` (early stopping)
- `batch=auto` (подгоняется под доступную память GPU)
- `optimizer=auto` (Ultralytics выбирает AdamW для малых моделей)
- Единый `seed=42`, single GPU (либо CPU для маленьких моделей).

## 5. Метрики и протокол оценки

**Точность (на валидации и тесте):**
- mAP@0.5
- mAP@0.5:0.95
- Precision (macro)
- Recall (macro)
- Per-class метрики (таблица)
- Confusion matrix

**Скорость:**
- FPS на GPU (если доступен)
- FPS на CPU (M1/M2 или Intel) — важно для «лёгкой» системы
- Latency p50/p95 на одну картинку
- Измеряется отдельно: preprocess / inference / postprocess

**Компактность:**
- Количество параметров (M)
- FLOPs (G)
- Размер весов на диске (MB)
- Размер экспортированной ONNX-модели

**OOD-обобщение:**
- mAP@0.5 на втором датасете без дообучения.

**Итог сравнения:**
- Сводная таблица (CSV + Markdown)
- Plot: mAP vs FPS, mAP vs params — для визуальной защиты
- Рекомендация на основе совокупного балла.

## 6. Real-time демо

**Функциональность:**
- Вход: веб-камера / видеофайл / папка с кадрами (CLI-флаг).
- Выход: окно OpenCV с боксами, классами, уверенностью.
- Оверлей: счётчики по классам, общее число обработанных кадров, текущий FPS.
- Опционально: трекинг (ByteTrack через Ultralytics) — чтобы одну и ту же бутылку не считать дважды.
- CSV-лог: timestamp, track_id, class, confidence, bbox.

**Контроль:** `q` — выход, `r` — сброс счётчиков, `s` — скриншот.

## 9. Риски и mitigation

| Риск | Mitigation |
|---|---|
| Roboflow-датасет окажется слишком маленьким / несбалансированным | Во второй итерации объединяем с другим Roboflow-датасетом; документируем ограничения в thesis. |
| Нет GPU — обучение на CPU слишком долгое | Google Colab free tier (T4) или Kaggle Notebooks. В скриптах явная поддержка `--device` флага. |
| Классы в датасете не совпадают с целевой таксономией | Работаем с тем, что есть в разметке; таксономия в работе описывается как «по факту данных». |
| Модели v11 требуют свежий ultralytics | Фиксируем версию в requirements.txt (>= 8.3.0). |
